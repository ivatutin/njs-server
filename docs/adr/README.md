# Architecture Decision Records (ADR)

В этой папке зафиксированы **архитектурные решения** проекта — что и почему было выбрано, какие альтернативы рассматривались, и какие последствия следуют.

## Зачем

- **Историческая память.** Через год кто-то спросит: «почему мы используем Prisma, а не TypeORM?» — ответ в ADR, не нужно полагаться на память
- **Онбординг.** Новый разработчик понимает не только *что* сделано, но и *почему*
- **Дисциплина выбора.** Когда решение записывается, оно обдумывается серьёзнее. Снижает рандом архитектурных решений
- **База для пересмотра.** Если контекст меняется (например, появились новые требования к скейлингу) — ADR содержит критерии «когда пересмотреть»

## Формат

Используем упрощённую версию [Michael Nygard format](https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions):

```
# ADR NNNN: Заголовок (что выбрано)

- Дата
- Статус: Proposed | Accepted | Superseded | Deprecated

## Контекст
Какая проблема, какие требования.

## Решение
Что мы выбрали.

## Последствия
Что хорошего, что плохого, mitigation для проблем.

## Альтернативы
Что рассматривали, почему отказались.

## Ссылки
External resources.
```

## Правила работы с ADR

1. **Одно решение = один файл.** Не пихать несколько решений в один документ.
2. **Имя файла:** `NNNN-kebab-case-title.md` с **нумерацией с ведущими нулями** (`0001-`, `0002-`, ...) — для естественной сортировки.
3. **После статуса `Accepted` файл становится immutable.** Если решение пересматривается:
   - Создать **новый** ADR с противоположным решением (или с уточнением)
   - В старом ADR поставить `Status: Superseded by ADR NNNN`
4. **Размер:** 1-3 страницы. Если получается больше — это не ADR, а проектная документация.
5. **PR ADR создаётся отдельно от кода.** Решение обсуждается, потом реализуется.

## Когда писать ADR

✅ **Писать ADR для:**
- Выбора фреймворка / ORM / IdP / message broker'а
- Архитектурного стиля (монолит vs микросервисы, событийная модель)
- Существенных правил, влияющих на структуру проекта (defense in depth, hexagonal layers)
- Решений, которые **сложно откатить** (миграция данных, выбор хранилища)
- Решений с компромиссом (мы выбрали X, потеряли Y — почему это ОК)

❌ **НЕ писать ADR для:**
- Стилистики кода (это в `.eslintrc` и CONTRIBUTING)
- Имён переменных
- Минорных рефакторингов
- Очевидных решений без альтернатив (используем git, используем PostgreSQL)
