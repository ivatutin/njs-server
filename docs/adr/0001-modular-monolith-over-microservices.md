# ADR 0001: Modular Monolith вместо микросервисов

- **Дата:** 2026-04-10
- **Статус:** Accepted
- **Решающие:** инициатор проекта

## Контекст

Нужно выбрать архитектурный стиль для нового backend-приложения. На входе: типовые ожидания продукта (несколько бизнес-доменов — пользователи, аутентификация, в перспективе биллинг/нотификации), команда из 1-3 разработчиков, нет жёстких требований к независимому скейлингу отдельных частей.

Доступные стили:

1. **Layered monolith** (классический trio controllers/services/entities)
2. **Modular monolith** (один процесс, изолированные модули по bounded contexts)
3. **Microservices** (отдельные сервисы, отдельные БД, HTTP/AMQP между ними)
4. **Serverless** (FaaS)

## Решение

Выбираем **Modular Monolith**.

Один деплоимый артефакт (Docker-образ), но внутри — независимые bounded contexts (`modules/user/`, `modules/auth/`) со своей доменной моделью, своим хранилищем (логически — PostgreSQL schemas), без прямых импортов между модулями.

Коммуникация между модулями — **только через domain events** на in-memory event bus (`InMemoryEventBus`), который абстрагирован за интерфейс `EventBus` и при необходимости заменяется на RabbitMQ.

## Последствия

### Положительные

- **Простота операционная.** Один процесс, один образ, один deployment. Логи, метрики, трейсы в одном месте.
- **Низкая стоимость инфраструктуры.** Один runtime, одна БД (с PostgreSQL schemas как поддоменами). Нет затрат на service mesh, message broker (пока в нём нет необходимости).
- **Атомарность транзакций.** Внутри bounded context можно использовать ACID без двухфазных коммитов.
- **Скорость разработки.** Можно переходить между модулями без переключения runtime, переинтеграции, обновления контрактов API.
- **Готовность к разделению.** Bounded contexts с чёткими границами и event-driven коммуникацией — это **тренировочная база для будущего разделения на сервисы**, если такой момент настанет.

### Отрицательные

- **Невозможность независимого скейлинга** конкретного модуля. Если User-функционал получит 10× нагрузку, скейлится весь процесс. Mitigation: горизонтальное масштабирование всего приложения покрывает первые 1-2 порядка нагрузки.
- **Нет физической изоляции** на уровне процессов. Утечка памяти в одном модуле убивает всё приложение. Mitigation: process supervisor (Docker restart policy), мониторинг.
- **Команды могут случайно нарушить изоляцию.** Соблазн `import` из соседнего модуля. Mitigation: code review, опционально — ESLint правило, опционально — Dependency-Cruiser для статического анализа графа зависимостей.
- **Технический долг при переходе на микросервисы** будет существенный — нужно вытаскивать в отдельные сервисы (требуется outbox, sagas, distributed tracing).

## Альтернативы

### Layered monolith (отвергнуто)

`controllers/`, `services/`, `entities/` на верхнем уровне.

**Почему нет:** при росте проекта фичи **размазываются по всем папкам** — изменение в `User` затрагивает `controllers/users/`, `services/users/`, `entities/users/`. Структура не отражает доменную модель. Нет естественной точки роста для bounded contexts.

### Microservices (отвергнуто)

Отдельные сервисы для User, Auth, Billing.

**Почему нет:**
- Для команды 1-3 человека overhead огромен: distributed tracing, retry/circuit breakers, schema evolution, eventual consistency
- Нет реальной потребности в независимом релиз-цикле или скейлинге
- На раннем этапе границы между bounded contexts ещё уточняются — менять API между сервисами дороже, чем менять интерфейсы внутри монолита
- **«Microservices premium»** оправдан только при доказанной потребности

### Serverless / FaaS (отвергнуто)

Functions как primary unit.

**Почему нет:**
- Сложные доменные модели плохо ложатся на cold-starts и stateless functions
- Связность функций через очереди превращается в неявную событийную архитектуру без чётких границ
- Vendor lock-in (AWS Lambda / Azure Functions API)
- Локальная разработка сложнее (эмуляторы)

## Когда пересмотреть

Это решение пересматривается, если возникнет **хотя бы одно**:

1. Один из модулей получает нагрузку в 10× выше остальных и стенки горизонтального скейлинга всего приложения дороже выделения сервиса
2. Команда вырастает до 10+ человек и параллельные релизы становятся узким местом
3. Появляется требование к разной технологической базе для разных доменов (например, ML-инференс на Python для одного модуля)
4. Регуляторные требования требуют физического разделения данных (например, GDPR-зона vs внешний рынок)

До этого — bounded contexts и event-driven коммуникация остаются достаточными.
